# 📚 听课笔记：缓冲区溢出攻击与系统级防护机制

## 📌 核心概述

本期内容聚焦 **栈缓冲区溢出（Stack Buffer Overflow）** 的成因、危害及利用原理，并系统讲解现代操作系统与编译器协同实现的三大防护机制：**栈随机化（ASLR）**、**栈保护者（Stack Canary）**、**不可执行位（NX/DEP）**。这些机制无需程序员手动介入，即可显著提升程序运行时安全性。

---

## 🔹 一、 缓冲区溢出原理与危害

### 1. 问题根源：C 语言无边界检查

> 💡 **相关笔记**：栈的布局和管理机制详见 [[3.7 过程（函数）的底层机制与栈帧管理]]

- C 语言数组访问**不进行运行时边界校验**，越界写入会直接覆盖相邻内存。
- 栈上布局（高地址 → 低地址）：
  ```
  [局部变量/缓冲区] ← 栈顶（低地址）
  [保存的寄存器]
  [返回地址] ← 关键控制信息
  [调用者栈帧...]
  ```
- **危险场景**：向栈上缓冲区写入超长数据 → 覆盖返回地址 → `ret` 指令跳转到攻击者指定位置 → 执行恶意代码。

### 2. 示例：`gets()` 函数的经典漏洞

```c
void echo() {
    char buf[8];      // 栈上分配 8 字节
    gets(buf);        // ❌ 无长度限制！输入 >7 字符即溢出
}
```

- **汇编视角**：编译器可能为 `buf` 分配 24 字节栈空间（含对齐/预留），但 `gets` 仍会无限制写入。
- **溢出阈值**：
  - 输入 ≤ 23 字节：可能仅覆盖局部填充区，程序侥幸运行
  - 输入 ≥ 24 字节：**返回地址被覆盖**，程序控制流劫持

> ⚠️ **历史教训**：1988 年莫里斯蠕虫、2001 年 Code Red 等著名攻击均利用此类漏洞。

## 🔹 二、 防护机制一：栈随机化（Stack Randomization / ASLR）

### 1. 核心思想

- **打破地址可预测性**：每次程序运行时，栈（及堆、库、代码段）的基地址随机变化。
- 即使攻击者掌握漏洞代码，也无法预先确定返回地址应覆盖为何值。

### 2. 实现细节（Linux x86-64）

- 属于 **地址空间布局随机化（ASLR, Address Space Layout Randomization）** 的子集。
- 栈地址随机范围约 2^24 个可能位置（具体熵值取决于内核版本与配置）。
- 验证代码：
  ```c
  int main() {
      int local;
      printf("%p\n", &local);  // 每次运行输出不同
  }
  ```

### 3. 安全收益

| 攻击场景     | 无 ASLR               | 启用 ASLR                           |
| :----------- | :-------------------- | :---------------------------------- |
| 本地溢出攻击 | ✅ 易成功（地址固定） | ❌ 需爆破地址（难度↑↑）             |
| 远程批量攻击 | ✅ 一台攻破，万台沦陷 | ❌ 每台地址不同，攻击成本指数级上升 |

> 💡 **注意**：ASLR 需配合 **PIE（Position Independent Executable）** 编译选项才能随机化代码段。

## 🔹 三、 防护机制二：栈保护者（Stack Canary / 金丝雀值）

### 1. 设计灵感

- 源自煤矿用**金丝雀检测毒气**：若金丝雀死亡（值被修改），说明栈已遭破坏，立即终止程序。

### 2. 工作流程

```
函数入口
   ↓
从只读段 %fs:40 读取随机 Canary 值
   ↓
将 Canary 压入栈，紧邻缓冲区高地址侧
   ↓
执行函数体
   ↓
返回前检查 Canary
   ↓
值不变？ ──是──→ 正常 pop 返回
   ↓
   否
   ↓
调用 __stack_chk_fail 终止程序
```

### 3. 汇编实现关键

```asm
; 函数入口：插入 Canary
mov    %fs:0x40, %rax     ; 从线程局部存储读取随机值
mov    %rax, -8(%rbp)     ; 存入栈上（缓冲区与返回地址之间）

; 函数返回前：验证 Canary
mov    -8(%rbp), %rax
xor    %fs:0x40, %rax     ; 与原始值异或
jne    __stack_chk_fail  ; 不等则调用错误处理
```

- **安全关键点**：
  - Canary 值**每次运行随机生成**，攻击者难以预测
  - 源地址 `%fs:0x40` 属于**只读线程局部存储**，用户态代码无法直接写入

### 4. 启用方式

- GCC 默认开启（`-fstack-protector-strong`）
- 可手动控制：`-fstack-protector-all` / `-fno-stack-protector`

## 🔹 四、 防护机制三：不可执行位（NX / DEP）

### 1. 攻击背景：注入可执行代码

- 传统攻击流程：溢出栈 → 写入 shellcode → 覆盖返回地址指向栈 → 执行恶意代码。
- 前提：栈内存页**同时具备可读、可写、可执行**权限。

### 2. 硬件级解决方案：NX Bit（No-eXecute）

| 时代        | 内存页权限模型                    | 缺陷                                   |
| :---------- | :-------------------------------- | :------------------------------------- |
| 早期 x86    | 可读/可写 合并为 1 位             | 可读即可能可执行，栈数据天然"可执行"   |
| 现代 x86-64 | **可读 / 可写 / 可执行 三态独立** | 栈可标记为 `RW-`（可读可写但不可执行） |

- **硬件支持**：CPU 页表项增加 **NX 位**（AMD） / **XD 位**（Intel）
- **操作系统配合**：链接器将 `.stack` 段标记为 `PROT_READ | PROT_WRITE`，**不含 `PROT_EXEC`**

### 3. 执行流程

```
攻击者尝试执行栈上代码
        ↓
CPU 取指时检查页表 NX 位
        ↓
若页标记为不可执行 → 触发 #PF 页故障（Page Fault）
        ↓
操作系统终止进程，攻击失败
```

> ✅ **优势**：硬件级检查，**零性能开销**；与 ASLR/Canary 组合形成纵深防御。

## 🔹 五、 三重防护的协同与局限

### 1. 防御矩阵效果

| 机制       | 阻止的攻击阶段          | 是否需代码修改    |
| :--------- | :---------------------- | :---------------- |
| **ASLR**   | 地址预测 → 跳转目标定位 | ❌ 编译器+OS      |
| **Canary** | 溢出检测 → 返回前拦截   | ❌ 编译器插入代码 |
| **NX**     | 代码注入 → 执行阶段拦截 | ❌ 硬件+OS+链接器 |

- **组合效果**：即使单点被绕过（如信息泄露破解 ASLR），其他机制仍可提供保护。

### 2. 现实局限性（攻击者仍在进化）

- **ROP（Return-Oriented Programming）**：不注入新代码，而是复用程序中已有的"小片段指令（gadgets）"构造攻击链，可绕过 NX。
- **信息泄露漏洞**：先读取 Canary/代码地址，再实施精准溢出。
- **面向数据的攻击**：不修改控制流，而是篡改关键数据（如权限标志）。

> 🔐 **安全本质**：没有银弹。防护机制提升攻击成本，但**安全编码实践**（如用 `fgets` 替代 `gets`、使用静态分析工具）仍是第一道防线。

## 📝 核心考点 & 面试高频题

1. **缓冲区溢出三要素**：无边界检查 + 栈布局 + 控制流劫持（返回地址覆盖）。
2. **Canary 检测时机**：函数**返回前**验证，而非写入时（避免误杀合法长字符串）。
3. **NX 位作用层级**：硬件页表检查，**非操作系统软件模拟**，故无性能损失。
4. **ASLR 随机化范围**：不仅栈，还包括堆、mmap 区域、共享库、可执行文件基址（需 PIE）。
5. **为什么 `%fs:0x40` 安全**：线程局部存储（TLS）的只读段，用户代码无法直接写入。
6. **纵深防御思想**：单一机制可被绕过，组合使用显著提升攻击门槛。
