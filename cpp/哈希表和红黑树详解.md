## 哈希表

### 什么是哈希表？

哈希表（Hash Table）是一种数据结构，它可以快速地存储和查找数据。你可以把它想象成一个超级快的电话簿：

```
传统电话簿：要找"张三"的电话，需要从头开始翻页
哈希表：直接跳到"张三"应该所在的页，一步到位！
```

### 核心概念

#### 1. 键值对（Key-Value Pair）

哈希表存储的是**键值对**：

```
键（Key）      值（Value）
"apple"  →   "苹果"
"banana" →   "香蕉"
"orange" →   "橙子"
```

- **键**：用来查找的东西（就像字典中的词条）
- **值**：实际存储的数据（就像词条的解释）

#### 2. 哈希函数（Hash Function）

哈希函数是一个神奇的转换器，它把任意输入变成一个数字（哈希值）：

```
哈希函数 f(x)
f("apple")   = 5
f("banana")  = 8
f("orange")  = 2
```

好的哈希函数有以下特点：
- **确定性**：相同输入总是产生相同输出
- **均匀分布**：不同输入尽可能产生不同输出
- **快速计算**：计算速度要快

#### 3. 哈希冲突（Hash Collision）

当两个不同的键产生相同的哈希值时，就发生了**冲突**：

```
f("cat")    = 3
f("dog")    = 3  ← 冲突！
```

解决冲突的常见方法：

**方法1：链地址法（Chaining）**
```
位置3：cat → dog → bird
       （用链表连接冲突的元素）
```

**方法2：开放地址法（Open Addressing）**
```
位置3：cat
位置4：dog  （找到下一个空位存放）
```

### 哈希表的操作

#### 1. 插入（Insert）

```
插入 "apple" → "苹果"

步骤：
1. 计算哈希值：f("apple") = 5
2. 去位置5看看
3. 如果位置5为空，直接存放
4. 如果位置5已有元素（冲突），使用冲突解决方法
```

#### 2. 查找（Search）

```
查找 "apple"

步骤：
1. 计算哈希值：f("apple") = 5
2. 直接去位置5
3. 找到了！返回 "苹果"
```

#### 3. 删除（Delete）

```
删除 "apple"

步骤：
1. 计算哈希值：f("apple") = 5
2. 去位置5
3. 找到并删除
```

### 时间复杂度

```
操作    最好情况    平均情况    最坏情况
插入    O(1)       O(1)       O(n)
查找    O(1)       O(1)       O(n)
删除    O(1)       O(1)       O(n)
```

**O(1) 的含义**：无论数据量多大，操作时间都差不多（常数时间）

**O(n) 的含义**：最坏情况下，所有数据都冲突，需要遍历所有元素

### 实际例子

想象一个简单的哈希表，存储学生的成绩：

```
键：学生姓名    值：成绩
"小明"    →    95
"小红"    →    88
"小刚"    →    92
```

查找"小红"的成绩：
1. 哈希函数计算 f("小红") = 2
2. 直接去位置2
3. 找到：88

这比从头遍历所有学生要快得多！

### 哈希表的优缺点

**优点**：
- ⚡ 查找、插入、删除速度极快（平均O(1)）
- 📊 适合快速查找的场景

**缺点**：
- 🎲 不能保持元素的顺序
- 🔄 冲突会影响性能
- 💾 需要额外的空间处理冲突

## 红黑树

### 什么是红黑树？

红黑树是一种**自平衡二叉搜索树**。它是二叉树的一种特殊形式，通过颜色标记和规则来保持树的平衡。

#### 基础概念：二叉搜索树（BST）

在理解红黑树之前，先了解二叉搜索树：

```
        50
       /  \
     30    70
    / \   / \
   20 40 60 80
```

**规则**：
- 左子树的所有节点 < 父节点
- 右子树的所有节点 > 父节点
- 每个节点最多有两个子节点（左和右）

**查找过程**：
```
查找 40：
1. 从根节点50开始
2. 40 < 50，去左子树（30）
3. 40 > 30，去右子树
4. 找到40！
```

#### 问题的产生：不平衡的树

```
正常情况：          不平衡情况：
      50                 10
     /  \                  \
   30    70                  20
                              \
                              30
                                \
                                40

平衡树：查找需要O(log n)
不平衡树：退化为链表，查找需要O(n)
```

### 红黑树的规则

红黑树通过5条规则保持平衡：

1. **每个节点要么是红色，要么是黑色**
2. **根节点是黑色**
3. **所有叶子节点（NIL）是黑色**
4. **红色节点的两个子节点必须是黑色**
   - 即：不能有两个连续的红色节点
5. **从任意节点到其每个叶子节点的路径包含相同数量的黑色节点**

### 红黑树的可视化

```
       B(50)          ← B=黑色
      /    \
   B(30)  B(70)
   /  \    /  \
 R(20)R(40)R(60)R(80)  ← R=红色
```

检查规则：
✓ 规则1：所有节点都有颜色
✓ 规则2：根节点50是黑色
✓ 规则3：叶子节点（未显示）是黑色
✓ 规则4：红色节点的子节点都是黑色
✓ 规则5：每条路径的黑色节点数量相同

### 红黑树的操作

#### 1. 旋转（Rotation）

当插入或删除导致不平衡时，通过旋转恢复平衡：

**左旋转**：
```
    P              Q
     \            / \
      Q    →    P   R
     / \          \
    S   R          S
```

**右旋转**：
```
      P          Q
     /          / \
    Q    →    R   P
   / \            /
  R   S          S
```

#### 2. 插入（Insert）

插入新节点的步骤：

```
1. 按照普通BST插入节点（新节点标记为红色）
2. 根据情况调整颜色和旋转
   - 情况1：父节点是黑色 → 无需调整
   - 情况2：父节点是红色 → 需要调整
```

**插入例子**：
```
原始树：
      B(50)
     /    \
   B(30)  B(70)

插入65（红色）：
      B(50)
     /    \
   B(30)  B(70)
         /
      R(65)  ← 新插入

检查规则：
- 65是红色，父节点70是黑色 ✓
- 无需调整
```

#### 3. 删除（Delete）

删除操作比插入更复杂，需要：
1. 删除节点
2. 调整树结构保持平衡
3. 可能需要多次旋转和变色

### 时间复杂度

```
操作    时间复杂度
查找    O(log n)
插入    O(log n)
删除    O(log n)
```

**O(log n) 的含义**：
- n=1000，约10步
- n=1000000，约20步
- n=1000000000，约30步

比线性查找O(n)快得多！

### 红黑树的优缺点

**优点**：
- 🎯 保证O(log n)的查找、插入、删除性能
- 📊 保持元素的有序性
- ⚖️ 自动平衡，无需手动维护

**缺点**：
- 🔧 实现复杂
- 📈 常数因子较大（比哈希表慢一些）
- 💾 需要额外的颜色信息

## 两者对比

### 性能对比

```
操作        哈希表        红黑树
查找        O(1)         O(log n)
插入        O(1)         O(log n)
删除        O(1)         O(log n)
有序遍历    ❌不支持      ✅支持
空间效率    较高         较低
实现复杂度  简单         复杂
```

### 使用场景

**使用哈希表的情况**：
- ✅ 需要快速查找（键值对查找）
- ✅ 不需要保持顺序
- ✅ 频繁插入和删除
- 例子：字典、缓存、去重

**使用红黑树的情况**：
- ✅ 需要保持元素有序
- ✅ 需要范围查询
- ✅ 需要遍历所有元素
- 例子：有序映射、区间查询、数据库索引

### 实际应用举例

**哈希表的应用**：
```
1. 编译器符号表
   - 快速查找变量名和函数名

2. 数据库索引
   - 某些数据库的哈希索引

3. 缓存系统
   - Redis的部分实现

4. JavaScript对象
   - obj["name"] = "Tom"
```

**红黑树的应用**：
```
1. C++ STL的map和set
   - 自动排序的关联容器

2. Linux进程调度
   - 完全公平调度器（CFS）

3. epoll实现
   - Linux的I/O事件通知机制

4. 数据库B+树
   - B+树是红黑树的变种
```

### 简单类比

**哈希表 = 快速存取柜**
```
你有1000个物品要存放
- 每个物品都有一个标签（键）
- 根据标签直接跳到对应的格子
- 不需要关心顺序，只要能快速找到就行
```

**红黑树 = 整齐的书架**
```
你有1000本书要排列
- 每本书都有一个编号
- 按照编号整齐排列
- 可以快速找到某本书，也可以遍历所有书
```

## 总结

### 选择指南

```
需要快速查找？                    → 哈希表
需要保持顺序？                    → 红黑树
需要范围查询？                    → 红黑树
内存有限？                        → 哈希表
实现复杂度要求低？                → 哈希表
需要稳定的最坏情况性能？          → 红黑树
```

### 关键要点

**哈希表**：
- ⚡ 平均情况下最快（O(1)）
- 🎲 最坏情况退化到O(n)
- 🚫 不支持有序遍历

**红黑树**：
- ⚖️ 始终保证O(log n)
- 📊 保持有序
- 🔧 实现较复杂

两者都是重要的数据结构，在实际开发中都有广泛应用。理解它们的特点和使用场景，能够帮助你做出更好的技术选择。
